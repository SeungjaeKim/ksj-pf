# EduFlow Pro CRM Drag Interaction Design

## Summary

`EduFlow Pro` 관리자 메인 보드의 상담 CRM에서 리드 카드를 단계 컬럼 사이로 직접 옮길 수 있는 상호작용을 추가한다. 목표는 포트폴리오 목업의 제품감을 높이면서도, 실제 운영툴처럼 자연스럽고 안전한 단계 이동 경험을 만드는 것이다.

이번 설계는 다음 원칙을 따른다.

- 데스크톱에서는 카드 드래그 앤 드롭이 기본 상호작용이다.
- 모바일과 좁은 화면에서는 드래그를 강제하지 않고, 버튼 기반 단계 변경을 유지한다.
- 일반 단계 이동은 즉시 반영한다.
- `등록 완료`, `보류 / 이탈`처럼 의미가 큰 단계는 드롭 후 확인을 한 번 더 받는다.

## Product Goal

현재 CRM 보드는 단계별 컬럼과 카드 배치가 이미 분리되어 있으므로, 드래그 이동을 붙이면 제품의 핵심 스토리인 `문의 → 상담 → 체험 → 등록` 흐름이 훨씬 더 직관적으로 보인다.

이번 변경으로 기대하는 효과는 다음과 같다.

- 원장이 칸반 보드처럼 리드를 직접 이동하는 실제 SaaS 감각 강화
- 컬럼 수치와 상세 패널이 동시에 변하는 연결형 데모 경험 강화
- 모바일에서는 대체 흐름을 유지해 접근성과 안정성 확보

## Interaction Model

### Desktop

- 각 리드 카드는 드래그 가능 상태를 가진다.
- 각 CRM 컬럼 본문은 드롭 존이 된다.
- 사용자가 카드를 컬럼 위로 드래그하면, 가능한 드롭 대상 컬럼이 시각적으로 강조된다.
- 드롭이 성공하면 해당 리드의 단계가 변경되고 즉시 보드가 다시 렌더링된다.

### Mobile / Narrow Width

- 데스크톱 드래그를 모바일에 억지로 확장하지 않는다.
- 리드 상세 패널 또는 카드 액션에 `다음 단계로 이동`, `단계 변경` 버튼을 둔다.
- 이 버튼은 동일한 상태 변경 함수를 호출하므로 데스크톱과 모바일의 상태 흐름은 동일해야 한다.

## Transition Rules

### Immediate Transitions

아래 단계는 별도 확인 없이 즉시 반영한다.

- `문의 접수`
- `상담 예정`
- `체험 예약`
- `체험 완료`

이동 직후 함께 바뀌어야 하는 항목은 다음과 같다.

- 카드가 새 컬럼으로 이동
- 컬럼 헤더 수치 갱신
- 우측 상세 패널의 현재 단계 문구 갱신
- 저장 상태 반영

### Confirmed Transitions

아래 단계는 드롭 후 확인 UI를 띄운다.

- `등록 완료`
- `보류 / 이탈`

확인 UI는 작고 빠른 인라인 모달 또는 오버레이여야 하며, 다음을 보여준다.

- 이동 대상 단계명
- 현재 선택된 리드 이름
- 확인 / 취소 버튼

취소 시에는 기존 단계가 유지된다.

## State Design

현재 `leadStages` 상태를 중심으로 단일 단계 변경 함수를 추가한다.

권장 상태 책임은 다음과 같다.

- `state-store.js`
  - 리드 단계 변경 함수
  - 확인이 필요한 단계 판별 함수
  - 임시 보류 중인 드래그 대상 상태
- `eduflow-pro.js`
  - 드래그 시작, 드래그 오버, 드롭, 확인/취소 이벤트 바인딩
  - 모바일 단계 변경 버튼 이벤트 바인딩
  - 상태 저장 후 CRM 재렌더
- `admin-view.js`
  - 카드와 컬럼에 드래그 관련 속성 추가
  - 확인 오버레이 및 모바일 대체 액션 UI 렌더

## Visual Feedback

드래그 상호작용은 아래 정도의 피드백을 준다.

- 드래그 중 카드에 `is-dragging` 스타일 적용
- 드롭 가능 컬럼에 `is-drop-target` 스타일 적용
- 확인 대상 단계는 드롭 후 즉시 이동하지 않고 확인 오버레이 표시
- 모바일 대체 버튼은 상세 패널 하단의 보조 액션으로 노출

비주얼 목표는 “과한 애니메이션”보다 “어디에 놓을 수 있는지 바로 이해되는 상태”다.

## Error Handling

- 알 수 없는 단계 ID로 드롭되면 무시하고 현재 상태 유지
- 존재하지 않는 리드 ID는 상태를 바꾸지 않는다
- 확인이 필요한 단계에서 사용자가 취소하면 아무 상태도 변경하지 않는다
- 브라우저 저장 실패 시에도 UI는 동작하지만, 다음 새로고침에서 상태가 유지되지 않을 수 있다

## Testing Strategy

`cscript.exe` 하네스 기준으로 검증 가능한 순수 동작을 우선 테스트한다.

- 리드 단계 직접 이동 함수가 즉시 단계 이동을 처리하는지
- `등록 완료`, `보류 / 이탈` 단계가 확인 필요로 판별되는지
- 드래그 렌더 마크업에 필요한 데이터 속성이 포함되는지
- 확인 오버레이 마크업이 렌더되는지
- 모바일 대체 액션 버튼이 렌더되는지

브라우저 수동 확인 항목은 다음과 같다.

- 데스크톱에서 카드 드래그 시 컬럼 강조
- 즉시 이동 단계 드롭 후 보드와 상세 패널 즉시 반영
- 확인 필요 단계 드롭 후 확인 UI 노출
- 모바일 폭에서 버튼 기반 단계 변경 가능

## Implementation Boundaries

이번 변경은 CRM 보드 단계 이동 경험에만 집중한다.

포함:

- 드래그 앤 드롭
- 단계 변경 확인 UI
- 모바일 대체 액션
- 상태 저장 및 하네스 검증

제외:

- 서버 연동
- 다중 선택 드래그
- 카드 정렬 순서 변경
- 드래그 이력 로그
- 복수 사용자 동시 편집

## Approval Note

이 설계는 사용자와 합의된 아래 기준을 반영한다.

- `1 + 3 결합`
- 데스크톱 드래그 중심
- 모바일/좁은 화면은 버튼 기반 대체 흐름
- 일반 단계는 즉시 이동
- `등록 완료`, `보류 / 이탈`은 확인 후 이동
